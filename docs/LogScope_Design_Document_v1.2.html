1.平台架構概覽

LogScope 由三個核心模組組成，彼此緊密整合：

模組說明
Log 分析器輸入 VRL 和 raw log，即時 parse 並顯示結果。核心體驗是左右分欄的即時循環：VRL 編輯 → parse → 結果預覽。
特徵庫 Library管理所有 vendor/product 的 log 特徵。支援手動建立和 LLM 自動爬取兩種建庫方式，並有 Review 機制控管品質。
LLM Copilot全平台感知的 AI 助手。根據當前頁面自動調整脈絡，提供 VRL 生成、log 解釋、Library 比對三個核心技能。

2.特徵庫 Library

Library 是整個平台的核心資產，採用三層結構：Vendor → Product → Log Type。

2.1 資料結構（ERD 摘要）

資料表層級重要欄位
VENDOR頂層name、slug、category、website_url、logo_url、status
PRODUCTVendor 下vendor_id、name、version、description、deploy_type、doc_url、status
LOG_TYPE核心表product_id、name、format、transport、status、source、reviewed_by、published_at
FIELD_SCHEMALog 欄位定義log_type_id、field_name、field_type、is_required、is_identifier、example_value、sort_order
PARSE_RULEVRL 版本log_type_id、vrl_code、version、status、created_by、updated_at
SAMPLE_LOGSample 集合log_type_id、raw_log、label（normal/edge_case/error）、added_by
LOG_TYPE.format 記錄原始格式（syslog、json、cef、leef、csv），直接影響 VRL 寫法。FIELD_SCHEMA.is_identifier 標記識別欄位（src_ip、user 等），供 Copilot 和比對引擎使用。PARSE_RULE 獨立成表以支援版本迭代和 diff review。

5.LLM Copilot

Copilot 的核心定位：不是通用聊天，而是懂 LogScope 脈絡的三技能專家。

5.1 三個核心技能

VRL 生成
輸入：raw log 或需求描述生成可執行 VRL；修正 parse 錯誤；解釋現有 VRL 每行
一鍵插入 Analyzer不需複製貼上，直接寫進編輯器
觸發方式：⌘K → "寫 VRL" · Analyzer 旁火花圖示
Log 解釋
輸入：一行或多行 log白話說明事件；標記異常值；對照 Library schema
多筆比較這幾筆有什麼差異
觸發方式：右鍵 → "解釋這行" · 結果欄選取後觸發
Library 比對
輸入：raw log識別 vendor/product（附信心% 和理由）；推薦 VRL rule
Library 沒有時建議觸發建庫；接受用戶糾正
觸發方式：Analyzer match bar 自動觸發 · 可手動問

5.3 全平台脈絡感知

頁面脈絡內容
Log 分析器目前 VRL 內容、已貼入的 log 筆數、Library 比對結果。可直接操作編輯器

7.給 Claude Design 的設計備忘

整體風格方向

參考現有 pyvrl-playground 的深色主題編輯器感，搭配乾淨的白色資訊區。Code 區塊用深色背景，非 code 區域用淺色系。整體偏向 developer tool 的專業感，而非消費者產品的活潑風格。

頁面設計重點
Library 列表頁卡片網格要能一眼看出三種狀態。未建庫的卡片建議用虛線邊框，讓用戶一眼看到「缺口」
Product 詳情頁Log Type 標籤切換是高頻操作，標籤列要易於點擊。右側 Copilot 建議卡片不能太顯眼，是輔助資訊而非主體
Log 分析器Library 比對列是「動作前提示」，建議放在三欄最上方。Parse 結果欄的欄位分組（識別/事件/數值）要視覺上有層次
Copilot panel脈絡列（context strip）是關鍵設計，讓用戶知道 AI 在看什麼。快速指令列的四個按鈕要夠顯眼，降低摩擦力
Review 頁面三欄 diff 視圖是整個建庫機制的體驗核心。左欄唯讀草稿、中欄可編輯、右欄即時試跑。低信心欄位要有明顯橘色警示

Tech Stack 確認

前端：Next.js
VRL 執行：參考 POC 專案 /Users/amos/Documents/side-projects/pyvrl-playground

8.1 後端架構

技術職責與備忘
uvPython 套件與虛擬環境管理。統一用 uv run 執行腳本，uv sync 同步依賴，避免 pip 混用問題
FastAPI
feature module		以功能模組切分 router：library/、analyzer/、pipeline/、copilot/。每個模組有獨立的 router.py、schemas.py、service.py，避免 God file。用 APIRouter prefix 統一掛載
PostgreSQL主資料庫，儲存 Library ERD 的六個核心表（VENDOR、PRODUCT、LOG_TYPE、FIELD_SCHEMA、PARSE_RULE、SAMPLE_LOG）。VRL code 和 sample raw_log 用 TEXT 儲存，field schema 的 metadata 可用 JSONB 擴充欄位。建議用 Alembic 做 migration
ClickHouse儲存 parse 執行的歷史紀錄與統計資料（哪些 VRL rule 被使用、parse 成功率、各 product 的使用頻率）。供 Library 頁面的使用統計和 Copilot 推薦依據。寫入為非同步 fire-and-forget，不擋主流程
Redis兩個用途：(1) LLM Pipeline job queue — 爬取、生成、發佈各步驟的狀態追蹤，用 Redis Stream 或搭配 ARQ/Celery。(2) API response cache — Library 詳情頁的欄位清單、VRL 不常變動，可 cache 降低 DB 壓力。TTL 設 5–15 分鐘，publish 時主動 invalidate

8.2 VRL 執行環境

POC 參考

現有 POC：pyvrl-playground 已有可動的 VRL parse 實作，LogScope 的 Analyzer 直接以此為基礎。開始前先讀懂 POC 的執行方式，再決定整合策略。

項目說明
執行方式確認先確認 POC 是用 Python subprocess 呼叫 VRL binary、還是 pyvrl binding，這決定後端 API 的設計。若是 Python binding，直接包成 FastAPI endpoint；若依賴 binary，需要處理 PATH 和 containerization
API 設計POST /analyzer/parse，body 帶 vrl_code 和 logs: list[str]，回傳每筆 log 的 parse 結果與錯誤訊息。單次請求上限建議 500 筆 log
前端 debounceNext.js 的 Analyzer 頁面：用戶停止輸入 VRL 或 log 後 400ms debounce 才觸發 API call，避免每個按鍵都打 API。parse 結果用 useTransition 標記為非緊急更新，不卡 UI
錯誤處理VRL compile error 和 runtime error 要分開回傳。Compile error 顯示在 VRL 編輯器行號旁；runtime error（某筆 log parse 失敗）顯示在結果欄對應那筆卡片上

8.3 前端架構

項目說明
Next.js App Router用 App Router（app/ 目錄）。Library 詳情頁和列表頁用 Server Component + generateStaticParams 預渲染常用 product；Analyzer 和 Copilot 因為有即時互動，用 Client Component
路由對應app/library/page.tsx → 列表頁、app/library/[vendor]/[product]/page.tsx → 詳情頁、app/library/[vendor]/[product]/review/page.tsx → Review 頁、app/analyzer/page.tsx、app/copilot/page.tsx
Copilot panel做成 layout-level component，掛在 app/layout.tsx 的右側，讓所有頁面共用同一個 Copilot panel 實例，對話狀態在頁面切換時保留。脈絡（context strip）靠 Zustand 或 React Context 全域管理，各頁面 mount 時 dispatch 自己的 context
VRL 編輯器建議用 CodeMirror 6（輕量、支援自訂 language extension）。可為 VRL 寫簡單的 syntax highlight extension，參考 POC 的關鍵字列表
API 呼叫用 fetch + SWR 或 TanStack Query 管理 server state。Library 資料 cache 在 client，Analyzer parse 結果不 cache（每次都是新的）

8.5 資料庫職責分工

資料庫儲存什麼不儲存什麼
PostgreSQLLibrary 六個核心表、user 資料、Pipeline job 狀態、Review 記錄log 原始資料、時序資料、大量寫入的統計
ClickHouseAnalyzer 的 parse 執行歷史（vrl_id、log_count、success_rate、latency_ms、ts）、Library 使用頻率統計需要 JOIN 的結構化業務資料、需要即時更新的資料
RedisPipeline job queue（Stream 或 List）、Library API response cache、Copilot 對話 session（TTL 1hr）需要持久化的業務資料
ClickHouse 的寫入建議 batch：Analyzer 每次 parse 完成後，把執行統計推入 Redis queue，由背景 worker 每 30 秒批次寫入 ClickHouse，避免高頻小批次寫入。

8.6 核心技術挑戰

挑戰建議策略
Library 比對演算法初期：把 raw log 前 200 字元送給 LLM，問它最可能是哪個 vendor/product（回傳 top 3 + 信心分）。中期：從 LOG_TYPE 的 FIELD_SCHEMA 和 sample 建立 fingerprint index，用特徵匹配提升速度和降低 LLM cost
LLM Pipeline 穩健性不同 vendor 文件格式差異大，爬取失敗或解析結果為空時要能 graceful fallback，並在 Review 頁面標示哪些欄位是「LLM 推測」而非「文件確認」。每個爬取步驟要有 retry + timeout
Copilot streaming用 FastAPI 的 StreamingResponse + Server-Sent Events（SSE），Next.js 前端用 EventSource 接收。這樣 Copilot 的回覆可以逐字顯示，不需等全部生成完才顯示
VRL POC 整合把 POC 的核心邏輯封裝進 features/analyzer/vrl_runner.py，對外暴露 parse(vrl_code: str, logs: list[str]) -> ParseResult 介面。其他模組只 import 這個介面，後續換實作不影響上層
